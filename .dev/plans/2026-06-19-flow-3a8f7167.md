### 3d. Helper: isWorkflowAllGreen (for a given set of excluded artifact paths)

Add a pure helper used inside `eligibleFirings` and `maintainDecisions`:

```ts
/**
 * Returns true when every artifact in `arts` is NOT a debt, excluding any
 * artifact whose path is in `excludePaths`. Used for the allGreen trigger's
 * bootstrap exclusion (the evaluator's own outputs are excluded from the check
 * so that 'all-green except for the evaluator itself' can be true).
 */
function allArtifactsGreen(arts: ArtifactMap, excludePaths: ReadonlySet<string>): boolean {
  for (const [path, a] of arts) {
    if (excludePaths.has(path)) continue;
    if (DEBT_STATES.has(a.acceptance)) return false;
  }
  return true;
}
```

### 3e. Extend eligibleFirings — allGreen branch

The current function iterates `def.loops` and handles `plain`, `map`, `reduce` modes. The default behaviour (`on: ['inputsGreen']` or `on` omitted) must be UNCHANGED.

**Insertion point:** at the top of the `for (const loop of def.loops)` body, before the existing `const mode = loopMode(loop)` code, add a dispatch on triggers:

```ts
const triggers = resolvedTriggers(loop);
const hasInputsGreen = triggers.includes('inputsGreen');
const hasAllGreen = triggers.includes('allGreen');
```

For the **inputsGreen path** — wrap the existing `if (mode === 'plain') ... if (mode === 'map') ... // reduce` logic in `if (hasInputsGreen)`. No other change to that logic.

For the **allGreen path** — a new block `if (hasAllGreen)` that runs INSTEAD of (or in addition to, but they're mutually exclusive in practice) the inputsGreen path:

```ts
if (hasAllGreen) {
  // Compute the set of output paths this evaluator loop produces.
  // These are excluded from the all-green check (bootstrap exclusion).
  const evaluatorOutputs = new Set<string>(
    plainOutputs(loop) // singletons + collection seals
  );

  // The workflow is all-green (excluding the evaluator's own outputs).
  const wfAllGreen = allArtifactsGreen(arts, evaluatorOutputs);
  if (!wfAllGreen) {
    // Workflow is not yet done — not eligible on this trigger.
    // (Still try inputsGreen path below if hasInputsGreen is also true — but
    // for the standard evaluator loop, on: is ['allGreen'] only.)
  } else {
    // Workflow IS all-green. Check if the evaluator still has a debt to discharge.
    // Use plain output check (an allGreen evaluator produces a singleton 'outcome').
    const outs = plainOutputs(loop).filter((p) => {
      const a = arts.get(p);
      return isDebt(a) && !frozen(a, loop);
    });
    if (outs.length > 0) {
      firings.push({
        loop: loop.name,
        key: '',
        inputs: [], // allGreen loop has no consumed inputs to fingerprint
        outputs: outs,
        cause: 'allGreen',
      });
    }
  }
}
```

**Key design details:**
- `inputs: []` — an allGreen loop is not gated on specific consumed inputs; by definition every artifact is green. Leaving inputs empty means the claim fingerprint is empty, which is correct (there is no specific input to CAS on).
- `cause: 'allGreen'` — threaded onto the Firing.
- `plainOutputs(loop)` computes singletons + seals — correct for a plain evaluator loop producing `outcome`.
- The exclusion uses `plainOutputs(loop)` (the evaluator's actual output paths), not the loop's consumed inputs. This is the bootstrap exclusion: if `outcome` is owed (a debt), it is excluded from the all-green check.

**Back-compat:** a loop with `on` omitted (or `on: ['inputsGreen']`) never reaches the `hasAllGreen` branch. `hasInputsGreen` is true and `hasAllGreen` is false, so existing logic runs exactly as before. ✓

### 3f. Extend maintainDecisions — allGreen re-arm

This is the fall-out-of-done mechanism: after `outcome` is green, if any other artifact becomes a debt (workflow is no longer all-green), the evaluator's `outcome` must re-arm.

**Mechanism (pure fingerprint approach):** The evaluator's `outcome` artifact, when green, stores a fingerprint. Instead of fingerpriinting specific input paths (it has no consumed inputs), we fingerprint a synthetic key — but fingerprinting nothing gives a trivially empty `{}`, which `fingerprintMatches` will always return true for (both empty), so the normal cascade will never re-arm it on its own.

Therefore we need a specific re-arm rule inside `maintainDecisions` for allGreen-triggered loops. **The rule:**

> For any artifact whose producer loop has `allGreen` in its triggers, if the artifact is currently **green** (and not terminal), and the workflow is **no longer all-green** (excluding that artifact from the check), then emit a structural `reject` op to re-arm it.

Insert this block BEFORE the main `for (const art of arts.values())` loop in `maintainDecisions`, or inside it as an additional branch:

**Preferred: add a new pass at the END of `maintainDecisions`** (after the existing loop), so the existing logic is untouched:

```ts
// allGreen re-arm: a green outcome of an allGreen-triggered loop is re-armed
// when the workflow is no longer all-green (excluding the evaluator's own outputs).
// This ensures the evaluator re-fires if the workflow later falls out of done.
for (const loop of def.loops) {
  const triggers = resolvedTriggers(loop);
  if (!triggers.includes('allGreen')) continue;

  // Compute this evaluator's output paths.
  const evaluatorOutputPaths = plainOutputs(loop);
  const evaluatorOutputSet = new Set<string>(evaluatorOutputPaths);

  // Check workflow all-green status excluding this evaluator's outputs.
  const wfAllGreen = allArtifactsGreen(arts, evaluatorOutputSet);
  if (wfAllGreen) continue; // workflow is all-green — no re-arm needed (stable)

  // Workflow is NOT all-green. Re-arm any green (non-terminal) evaluator output.
  for (const path of evaluatorOutputPaths) {
    const art = arts.get(path);
    if (!art || art.acceptance !== 'green' || art.terminal) continue;
    ops.push({
      kind: 'reject',
      path,
      reason: 'allGreen-rearm: workflow fell out of done — re-evaluating',
    });
  }
}
```

**No-thrash guarantee:** `allArtifactsGreen` is a pure function of `arts`. When the workflow IS all-green and `outcome` is owed (already a debt), this branch does nothing (`wfAllGreen` is true → `continue`). When `outcome` is green and the workflow IS all-green, this branch does nothing (same `continue`). When `outcome` is green and the workflow is NOT all-green, this emits one `reject` op. On the next pass (after the reject is applied), `outcome` is now a debt → not green → no op. Stable after one pass. ✓

## Step 6 — docs/design.md

Add a new section `§21` after `§20` (the effect contract section). Follow the same bullet-list style as §20.

```
## §21 Firing rules and the completion evaluator (`on:`)

Every loop today is implicitly `on: [inputsGreen]` — fire when consumed inputs are green. `on:` makes the firing trigger explicit.

- **§21.1 `inputsGreen` (default)** — the existing behaviour, unchanged. A loop whose `on:` is omitted, or explicitly set to `['inputsGreen']`, fires exactly as today.
- **§21.2 `allGreen`** — the loop fires when the workflow is all-green: no outstanding debts among all artifacts *except the evaluator's own produced outputs* (bootstrap exclusion). Fires immediately on all-green (no delay — the `idle` trigger, which waits, is a planned follow-up, PR3b).
- **§21.3 Bootstrap exclusion** — the evaluator's own owed `outcome` is not counted among the debts in the all-green check. Without this, the evaluator's firing could never be triggered (its own debt would prevent all-green).
- **§21.4 Fall-out-of-done re-arm** — once `outcome` is green (done), if the workflow later falls out of all-green (a new debt appears — e.g. a re-provided input re-arms an upstream artifact), `maintainDecisions` detects that `outcome` is green but all-green no longer holds, and emits a structural reject to re-arm `outcome`. When the workflow returns to all-green, `eligibleFirings` offers the evaluator again. This is stable: `maintainDecisions` only emits the op when the workflow is NOT all-green but `outcome` IS green. After the reject is applied, `outcome` is a debt — the op is not re-emitted.
- **§21.5 Trigger-cause** — the engine threads the cause ('allGreen') onto the `Firing`, the `RunData`, and the `Order`. A worker can read `order.cause` to branch behaviour (e.g. inspect status, green `outcome`, message a human).
- **§21.6 One `outcome` output** — the evaluator loop produces exactly one singleton `outcome` artifact. This is the embedding boundary contract (§17): the outer workflow or teardown step consumes the child's `outcome`.
- **§21.7 The `idle` trigger** — deferred to PR3b. Any `on:` token other than `inputsGreen` or `allGreen` is a hard `validateDef` / `buildLoop` error pointing at the PR3b follow-up.
```

## Edge cases to be explicit about

1. **allGreen + no other loops (pure evaluator workflow):** a workflow with ONLY the completion evaluator and no other loops has no artifacts to be debts except `outcome` itself. Bootstrap exclusion makes `allArtifactsGreen` return true immediately. The evaluator fires at T=0. This is correct — an empty graph is done.

2. **allGreen loop also in inputsGreen:** `on: ['inputsGreen', 'allGreen']` is technically valid (both tokens are supported). The builder does NOT need to handle this — it is an unusual but valid def. The implementation naturally handles it: `hasInputsGreen` is true so the existing logic runs; `hasAllGreen` is also true so the allGreen block runs. The firing deduplication is by (loop, key, outputs) — the same loop would produce two Firing objects with the same loop+key+outputs but different `cause`. Since the plan calls for one evaluator wired ONLY to allGreen, the builder can assume `on: ['allGreen']` is the common case and test only that. No special dedup logic is required (the engine's existing cadence/parallel guards handle double-firings).

3. **inputs: [] in eligibleFirings for allGreen:** the claim fingerprint will be empty (`{}`). `fingerprintMatches(arts, [], {})` — let's verify: `paths.length === 0`, `keys.length === 0`, both are 0 → returns true. A born-reject happens when `!fingerprintMatches` at commit time. With an empty fingerprint, the CAS always passes, meaning an allGreen firing is never born-rejected for input movement. This is correct: there are no inputs to move.

4. **frozen evaluator:** if the evaluator's `outcome` has stalled (maxAttempts exceeded), `frozen(a, loop)` returns true, and `eligibleFirings` will NOT offer a new firing. This is intentional and consistent with the existing stall machinery — a human must `retry` to clear the stall.

5. **store.ts / insertRun and the `cause` column:** if the store uses individual SQL columns for RunData fields, adding `cause` requires a migration. Follow the same pattern as `sessionId` was added (check `src/store.ts` `insertRun` and the `runs` table DDL). If the runs table stores a JSON blob, no migration is needed. The builder must check before writing any migration.

6. **No src/model.ts I/O:** the entire model.ts change uses only `arts`, `def`, and pure functions. No `Date.now()`, no `now` parameter, no timers. The `allArtifactsGreen` helper is a pure Map iteration. ✓

7. **modelCheck / BFS compatibility:** `modelCheck` calls `eligibleFirings` and `settleInMemory`. After this PR, BFS will explore allGreen firings when the workflow is done. This is desirable — `completable` in the model checker will now also find the evaluator's outcome as a reachable completion. The `Firing.cause` field on BFS transitions is ignored by the checker (it looks at loop/key/outcome, not cause). No changes to `modelCheck` needed.

8. **`workflowStatus.done` definition unchanged:** `done = ![...arts.values()].some(a => DEBT_STATES.has(a.acceptance))`. This counts the evaluator's `outcome` as a debt if it is owed. That means `done` is false while `outcome` is owed. This is correct: the workflow is not done until the evaluator has spoken. Once `outcome` is green (the evaluator ran), `done` is true. This is the right semantics and requires no change.
